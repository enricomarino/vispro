Notazione XML

Le informazioni riguartante i vari widget del sistema sono descritte da un documento XML, definito da un documento XML Schema.

Si è scelto come documento di definizione lo standard XML Schema del W3C; rispetto al precedente DTD, uno schema può definire tipi di dato predefiniti (e.g., interi, stringhe, date ed orari), consente l’uso di tipi complessi definiti dall’utente, e può esprimere riferimenti all’interno di un file XML.

Formato del file XML

Il documento XML contiene un nodo dadice <visprogw>.

In nodo radice <visprogw> fa riferimento a due XML schema:

  • lo standard W3C;

  • lo schema XML per VisProGW, descritto in dettaglio nel seguito.

Il nodo radice <vispro> contiene un unico attributo:

  • <version>, un intero senza segno che indica la versione del documento.

e i seguenti nodi:

  • <types>: una lista di tipi di widget;

  • <widgets>: una lista di widget;

  • <source>: un sorgente (non validato), contenuto in un blocco CDATA.

Scheletro del file XML
<?xml version="1.0" encoding="UTF-8"?>

<!-- Start XML configuration -->
<vispro
    version="0"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xsi:noNamespaceSchemaLocation="vispro.xsd">

    <!-- Types list -->
    <types>
        <type name="..."/>
    </types>

    <!-- Widgets list -->
    <widgets>

        <widget>
            <!-- ... -->
        </widget>

    </widgets>

    <!-- HTML source code -->
    <source>
        <![CDATA[

        ]]>
    </source>
</vispro>

Definizione dei Tipi

Il nodo <types> della root <visprogw> contiene la lista dei tipi generici di widget definite nel documento. Il nodo <types> non contiene attributi, ma contiene uno o più nodi <type>.

Un nodo <type> contiene un unico attributo:

  • name: attributo obbligatorio di tipo xs:ID (univoco all’interno del documento XML) utilizzato per la tipizzazione delle widget, e per la definizione delle dipendenze tra widget, come descritto in seguito.

Esempio di Tipi
<types>
    <type name="zoom" />
    <type name="map" />
    <type name="search" />
    <type name="info" />
</types>

Ogni widget definisce un tipo specifico (un sotto-tipo) di uno dei tipi generici. Il tipo di una widget è definito dall’attributo type, mentre il sotto-tipo è definito dall’attributo name, come descritto in seguito.

Ad esempio, le widget 'zoom verticale' e 'zoom orizzontale' sono considerati tipi specifici del tipo generale zoom.

Definizione dei Widget

Il nodo <widgets> non contiene attributi, ma contiene uno o più nodi <widget>.

Un nodo <widget> contiene i seguenti attributi:

  • type: attributo obbligatorio di tipo xs:IDREF che identifica il tipo di widget;

  • name: attributo obbligatorio di tipo xs:ID che identifica il sotto-tipo di widget;

e i seguenti nodi:

  • <description>: nodo obbligatorio, conteniene una stringa che definisce una descrizione della widget;

  • <label>: nodo obbligatorio, conteniene una stringa che definisce una descrizione breve (una etichetta) della widget;

  • <dimensions>: nodo obbligatorio, contiene due nodi, <width> e <height>, che definiscono le dimensioni della widget, come descritto in seguito;

  • <images>: nodo obbligatorio, contiene due nodi, <icon> e <image> che definiscono le rappresentazioni della widget, come descritto in seguito;

  • <dependencies>: nodo obbligatorio, contiene zero o più nodi <dependency> che definiscono le dipendenze verso altre widget, come descritto in seguito;

  • <properties>: nodo obbligatorio, contiene una lista di elementi associati a proprietà della widget, come descritto in seguito;

  • <code>: nodo obbligatorio contiene uno dei tag insert per la definizione del codice da generare, come descritto in seguito.

L’attributo type, descritto sopra, essendo di tipo xs:IDREF, può assumere valori di tipo xs:ID. Per questo motivo, nel caso in cui l’attributo contenga una stringa che non compare nella definizone dei tipi (unico punto in cui compaiono elementi di tipo xs:ID), il documento XML non risulta valido e la sua validazione genera un errore. In questo modo, non è possibile definire widget appartenenti ad un tipo generico non specificato; inoltre, ancora più importante, non è possibile definire una dipendenza verso una widget di un tipo non specificato. Il meccanismo xs:ID/xs:IDREF permette di rilevare questo tipo di incongruenze al tempo di validazione del documento XML. :)

Scheletro di un widget
<widget type="zoom" name="zoom_h_s" >
    <description>Horizontal Scroll Zoom</description>
    <dimensions>
        <height value="168" min="10" max="300" resizable="true"/>
        <width value="26" resizable="false"/>
    </dimensions>
    <images>
        <icon url="http://.../zoom_h_s_ico.png" />
        <image url="http://.../zoom_h_s.png" />
    </images>
    <dependencies>
        <dependency type="map" name="reqmap" required="true">
            <description>La mappa richiesta dall'oggetto zoom</description>
        </dependency>
    </dependencies>
    <properties>
        <bool name="tooltip" label="questa è una proprietà di tipo booleano" default="true" />
    </properties>
    <code>
        <insert match="html">
            <![CDATA[
            <div id="<%= id %>" style="position:absolute; top:<%= top %>px; left:<%= left %>px; width:<%= width %>px; height:<%= height %>px; z-index:<%= zIndex %>"></div>
            ]]>
        </insert>
        <insert match="js">
            <![CDATA[
            var <%= id %> = geopoi.sdk.zoom("<%= id %>",
            {
                map: <%= reqmap %>,
                tooltip: <%= tooltip %>,
                vertical: false,
                slider: true,
                size: <%= width %>
            } );
            ]]>
        </insert>
    </code>
</widget>

Dimensions

Il nodo <dimensions> contiene i nodi <width> e <height> che definiscono le dimensioni della widget.

I nodi <width> e <height> contengono i seguenti attributi:

  • value: attributo obbligatorio, intero non negativo che definisce il valore della dimensione;

  • resizable: attributo _obbligatorio, booleano con default true, che indica se la dimensione è modificabile;

  • min: attributo opzionale, intero positivo che definisce il valore minimo della dimensione;

  • max: attributo _opzionale, intero positivo che definisce il valore massimo della dimensione.

Esempio di dimensions
<dimensions>
    <height value="168" min="10" max="300" resizable="true"/>
    <width value="26" resizable="false"/>
</dimensions>

Osserviamo che, nell’esempio, l’altezza della widget è di 168 (pixel), mentre la larghezza è di 26 (pixel); inoltre, l’altezza è ridimensionabile, e può assumere un valore compreso tra 10 e 300 (pixel).

Osserviamo che, in generale, l’unità di misura utilizzata è il pixel!

Images

Il nodo <images> contiene i nodi <icon> e <image> che definiscono le rappresentazioni della widget:

  • <icon>: nodo obbligatorio, definisce l’icona della widget.

  • <image>: nodo obbligatorio, definisce l’immagine della widget all’interno del campo di lavoro, ovvero una anteprima della widget quando l’appilcazione generata sar&agrave in esecuzione nel browser.

Entrambi i nodi <icon> e <image> contengono un attributo:

  • url: attributo obbligatorio, indica l’url dell’icona o dell’immagine.

Dependencies

Il nodo <dependencies> contiene zero o pi&ugrave nodi <dependency> che definiscono le dipendenze della widget.

Un nodo <dependency> contiene i seguenti attributi:

  • type: attributo obbligatorio, di tipo xs:IDREF, indica il tipo di widget che soddisfa la dipendenza;

  • name: attributo obbligatorio, stringa che indica il nome della dipendenza;

  • required: attributo obbligatorio, booleano che indica se la dipendenza deve essere soddisfatta obbligatoriamente affinché il widget possa funzionare correttamente.

e un nodo <description> che contiene una stringa che fornisce una descrizione testuale della dipendenza.

Esempio di dependency
<dependencies>
    <dependency type="map" name="reqmap" required="true">
        <description>La mappa richiesta dalla widget zoom</description>
    </dependency>
    <dependency type="info" name="reqinfo" required="false">
        <description>La info window richiesta dalla widget zoom</description>
    </dependency>
</dependencies>

Nell’esempio, la widget definita richiede necessariamente il collegamento con una widget di tipo generico map, tale dipendenza obbligatoria, con nome reqmap, ha una descrizione testuale 'La mappa richiesta dall’oggetto zoom'; inoltre, la widget definita può essere collegata con una widget di tipo generico info; tale dipendenza opzionale, con nome reqinfo, ha una descrizione testuale 'La info window richiesta dalla widget zoom'.

Properties

Il nodo <properties> contiene zero o più nodi <bool>, <number>, o <string>.

I nodi <bool>, <number>, o <string> contengono i seguenti attributi:

  • name: attributo obbligatorio, indica il nome della proprietà che potr&agrave essere utilizzata nella generazione del codice;

  • label: attributo obbligatorio, contiene una descrizione breve della proprietà;

  • default: attributo opzionale, specifica un valore di default della proprietà.

Inoltre, i nodi <number> contentono i seguenti attributi:

  • min: attributo opzionale, indica un valore minimo della proprietà;

  • max: attributo opzionale, indica un valore massimo della proprietà.

Proprietà predefinite

Ogni widget possiede sei proprietà intriseche:

  • id: una stringa identificativa della widget, generata dal sistema;

  • top: la coordinata verticale minima della widget (in pixel);

  • left: la coordinata orizzontale minima della widget (in pixel);

  • width: la larghezza della widget (in pixel);

  • height: l’altezza della widget (in pixel);

  • zIndex: lo z-index del widget (in pixel).

Tali proprietà possono essere utilizzate nella generazione del codice, come descritto in seguito.

La posizione della widget (definita dai valori top e left) è determinata a run-time. Le dimensioni della widget (definite dai valori width e height) sono soggette alle eventuali restrizioni specificate nel nodo <dimensions>. Lo zIndex della widget (definito dal valore zIndex) °rave; modificabile a run-time.

Insert

Il nodo <code>, tramite i nodi <insert>, specifica il codice che il sistema dovrà generare automaticamente.

Un nodo <insert> contiene un solo attributo:

  • match: attributo obbligatorio, di tipo stringa, indica il punto di inserzione all’interno del sorgente specificato nel nodo <source>, descritto in seguito.

e una stringa, dunque non validabile, che specifica il codice da generare.

Il sistema inserirà il codice della widget (ovvero la stringa contenuta nel nodo <insert> nella definizione della widget) all’interno del codice di output (ovvero la stringa contenuta nel nodo <source> del documento XML), in particolare, nel punto di inserzione specificato dall’etichetta con il valore di match.

I punti di inserzione nei codici sono definiti da etichette (stringhe semplici, senza spazi intermedi) racchiuse da i simboli <%= e %>.

Esempio di codice da generare
<widgets>
    ...
    <widget type="zoom" name="zoom_h">
        ...
        <code>
            <insert match="html">
                <![CDATA[
                    <div id="<%= id %>" style="top:<%= top %>px; left:<%= left %>px" ></div>
                ]]>
            </insert>
            <insert match="js">
                <![CDATA[
                var <%= id %> = geopoi.sdk.zoom("<%= id %>",
                {
                    map: <%= reqmap %>,
                    tooltip: <%= tooltip %>,
                    vertical: false,
                    slider: true
                } );
                ]]>
            </insert>
        </code>
    </widget>
    ...
</widgets>

<source>
    <![CDATA[
        ...
        <%= html %>
        ...
        <%= js %>
    ]]>
</source>

Nell’esempio, la widget definita produce due porzioni di codice, da inserire rispettivamente nel codice di output finale da generare nei punti di inserzione denominati 'html' e 'js' (<%= html %> e <%= js %>).

Tali porzioni di codice saranno sottoposte a sostitzioni.

La porzione di codice denominata 'html' (il testo contenuto nel nodo <insert match="html">) contiene tre punti di inserzione denominati rispettivamente 'id', 'top' e 'left' (<%= id %>, <%= top %> e <%= left %>), che verranno sostituiti rispettivamente con i valori delle proprietà della widget 'id', 'top' e 'left'. Ricordiamo che le proprieta id, top e left sono propriet&agrave intrinseche di una widget.

La porzione di codice denominata 'js' (il testo contenuto nel nodo <insert match="js">) contiene tre punti di inserzione denominati rispettivamente 'id', 'reqmap' e 'tooltip' (<%= id %>, <%= reqmap %>, <%= tooltip %>) che verranno sostituiti con i valori della proprietà 'id', 'reqmap' e 'tooltip' della widget. Definizione del codice

Il sistema genererà un codice sorgente a partire da un template specificato del tag source della root. Tale codice sarà sottoposto alla sostituzione dei tag tra parentesi angolate con segno d’uguaglianza.

Di seguito si riporta un esempio di codice sorgente template, in questo caso, un file HTML.

Esempio di codice
<source>
    <![CDATA[
    <html>
    <head>
        <title>Hello</title>
    </head>
    <body>
        <%= html %>
        <h1>Hello</h1>
        <p>This is a test</p>
        <%= js %>
    </body>
    </html>
    ]]>
</source>

Si ricorda che la root del file XML sarà l’elemento <vispro>.
